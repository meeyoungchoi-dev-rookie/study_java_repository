## 상속

- 기존 클래스를 재사용하여 새로운 클래스 작성
- 코드를 공통적으로 관리하므로 코드의 추가 및 변경이 쉽다
- 상속은 확장의 개념이다
- 왜?
- 상속받는 클래스의 멤버 개수가 계속 늘어난다

## 다형성

- 여러가지 형태를 가질수 있는 능력
- 객체의 역할이 상황에 따라 달라질수 있다
- 예)
- 할인 정책의 종류로 기간 할인 정책과 비용할인 정책이 있다
- 영화 입장에서는 구체적인 할인 정책이 무엇인지 중요하지 않다
- 영화가 할인 정책이 적용만 되 있으면 어떤 할인 정책인지는 내부에서 상황에 따라 달라진다
- 새로운 할인 정책이 생기는 경우 할인 정책 인터페이스를 구현하기만 하면 다양한 할인 정책을 만들어 낼 수 있다
- 영화에 적용된 할인 정책이 기간할인 정책이라면 해당 할인 정책 객체는 기간 할인 정책으로써 역할을 수행할 것이다
- 클라이언트는 구체적인 객체의 역할이 무엇인지 알필요 없다
- 구체적인 객체의 역할은 실행시점에 결정된다

- 상속과 인터페이스에 둘다 다형성이 적용될 수 있다
- 상속을 통해 기능을 확장해 나가는 것도 다형성이고 인터페이스를 통해 공통 기능을 구현하도록 강제하는것도 다형성이다
- 구현체가 무엇이든 관계없이 인터페이스 타입만 일치한다면 여러개의 구현체를 만들어 낼수 있다
- 즉 동일한 역할을 수행하는 객체라면 얼마든지 서로 대체가 가능하다

### 매개변수의 다형성

- 메서드의 파라미터를 인터페이스 타입으로 받을 수 있다
- 왜?
- 상속이든 인터페이스든 관계없이 클라이언트 입장에서 구현체가 구현한 인터페이스 타입이 중요하기 때문이다
- 내부 로직은 인터페이스 뒤로 캡슐화 시킬 수 있다

## 추상클래스

- 여러 클래스에서 공통으로 갖는 속성 또는 행위 한곳에 모아둔다
- 상속과 다른 점은 상속은 자손이 부모를 바탕으로 확장해 나간다
- 추상클래스는 자손에서 공통으로 사용하는 부분을 뽑아 부모 클래스로 만들어 둔것이다
- 즉 , 위로 올라갈수록 자손 클래스가 공통으로 사용하는 요소들만 남게 된다
- 공통된 책임을 갖는 객체의 데이터와 동작을 한곳에 모아두고 재사용이 가능하다

### 추상 메서드의 장점

- 자손들이 공통으로 사용하는 메서드의 선언부는 변하지 않는다
- 하지만 자손들의 상황에 맞게 메서드 내부 구현은 변경될 수 있다
- 추상 메서드를 통해 어떤 기능 구현을 위한 것인지만 알려주고 내부 로직은 자손들이 상황에 맞게 직접 구현한다
- 클라이언트 입장에서 메서드 내부 로직을 몰라도 호출할 수 있다
- 즉 , 내부 로직은 캡슐화 된다

## 인터페이스

- 추상 클래스보다 추상화 정도가 높다
- 객체들이 공통 책임을 가질때 객체의 역할을 모아놓은 집합
- 즉 서로 관계가 없는 클래스에게 관계를 맺어 준다
- 예)
- 할인 조건의 종류로 기간할인과 비용할인이 있는데
- 두개의 클래스는 서로 관련이 없다
- 단지 할인 조건을 만족하는 경우 할인 비용을 계산하는 공통된 역할만 수행한다
- 역할을 인터페이스로 분리하여 내부 로직을 몰라도 공통된 역할을 수행하도록 추상화 시킬 수 있다
- 즉 다형성이 적용되게 된다
- 또한 내부 구현은 드러나지 않으므로 외부로부터 캡슐화 된다

## 언제 추상 클래스를 사용하고 언제 인터페이스를 사용해야 하는가

- 추상 클래스 - 부모 클래스의 기능을 재사용 하거나 확장하는 경우
- 인터페이스 - 인터페이스를 구현한 객체들이 동일한 동작을 해야 할때

## 결론

- 객체들간 의존성을 낮춘다
- 코드의 변경에 빠르게 대처할 수 있다
- 불필요한 내부 구현은 숨길수 있다
    - 응집도가 높아지고 결합도가 낮아진다
- 객체들이 서로 협력을 더 잘 할 수 있게 된다
- 서로 동일한 행동을 하는 객체라면 구체적인 역할은 다형성을 통해 대체될 수 있다
